Push Notifications

A design preview and technical overview for browser push notifications — the future system that will alert BMFFFL managers to key league events in real time.

🔔

What's Coming

Push Notifications — Coming in Season 7

BMFFFL is planning native browser push notifications for key league events. This means you'll get pinged — even with the tab closed — when a trade drops, a waiver claim clears, the Commissioner posts, or your lineup has a problem. No more refreshing Sleeper every five minutes on trade deadline day.

Planned — Not Yet Built

Section 3

How It Would Work

1

Browser asks permission

On first visit (or when you click "Enable Notifications"), the browser displays a native permission dialog. You can allow or deny — no permission, no push.

2

Service worker registers

A small background script (service worker) is installed in your browser. It stays resident even when the site tab is closed, ready to receive push messages.

3

Sleeper webhook fires

When a trade is processed, a player is dropped, or another key event happens in the league, Sleeper POSTs a signed payload to our server endpoint.

4

Our server sends push

The server verifies the webhook signature, determines which owners should be notified, and fans out Web Push messages to each subscribed browser.

Browser Support

  • Chrome (Android & Desktop)
  • Firefox
  • Edge
  • Safari — iOS 16.4+ & macOS Ventura+
  • Safari — iOS 16.3 and below

Section 5

Implementation Requirements

Server component

Can't do this with static export alone — requires a live server endpoint to receive Sleeper webhook POSTs.

Vercel serverless / edge function

A single `/api/webhook` route deployed as a Vercel Function handles both Sleeper events and Web Push dispatch.

Web Push API + VAPID keys

VAPID (Voluntary Application Server Identification) key pair signs push messages so browsers can verify our identity.

Sleeper webhook integration

Commissioner registers our webhook URL in Sleeper settings. Sleeper signs each payload with a shared secret.

Subscription storage

Each browser push subscription endpoint must be persisted (e.g., Vercel KV or a small database) so the server knows where to send messages.

⏱️

Estimated Effort

1–2 sprints — assuming Vercel deployment, existing Sleeper integration, and a simple KV store
